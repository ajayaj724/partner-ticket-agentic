What is an agent?

The leap from calling an LLM to building an agent is the conceptual centre of this whole project. The word "agent" is overloaded in industry; let's nail down what we mean by it.

The shortest possible definition

An agent is a small piece of software that:

  1. Has a goal (one specific job to do)
  2. Uses an LLM to make decisions about that job
  3. Optionally uses tools (functions that fetch data or take actions in the outside world)
  4. Returns a structured result

That's it. An "agent" in our system isn't AGI; it's a focused function with an LLM at its heart and tools at its hands.

If you know your shipped Quarkus project

Each of your Quarkus Lambdas had one job (talk to downstream system A, B, or C). It had a typed input, typed output, and a focused responsibility. An agent in our system is shaped the same way — except instead of a hardcoded business rule, an LLM decides what to do given the input, and the output is Pydantic-validated like a Quarkus DTO would be.

Why call it an "agent" and not just "a function"?

Three things distinguish an agent from a regular function:

1. It uses an LLM in its decision-making

A regular function has hardcoded logic. route(ticket) is a chain of if/else statements. An agent uses an LLM to handle the part of the decision that's hard to express in if/else. For example, our F1 Triage agent uses an LLM to classify free-text into a category — because writing if/else for every possible way a partner might phrase "my circuit is down" is a losing battle.

2. It can use tools dynamically

An agent doesn't have to call all its tools every time. It can look at the input, decide which tools are relevant, and call only those. (Our F2 Enricher technically could work this way; today it calls all 4 tools every time because the surface is small — but the architecture supports dynamic tool selection.)

3. It exposes a typed contract

An agent's job is to always return a specific shape, no matter what the LLM said. The contract is the Pydantic schema:

F1 Triage's typed contractclass TriageOutput(BaseModel): category: Literal["circuit_down", "throughput_degraded", "appointment_request", "billing", "provisioning", "other"] urgency: Literal["critical", "high", "medium", "low"] entities: Entities confidence: float # 0.0 - 1.0 rationale: str

If the LLM returns something that doesn't fit this shape, the agent rejects it with a ValidationError rather than passing the mess downstream. That's what "schema-first" means.

What makes a system "agentic"?

"Agentic AI" is the industry term for a system built out of multiple agents that work together — usually with some orchestration logic deciding which agent runs when, in what order, and what state to pass between them.

The opposite of "agentic" would be:

  • A single LLM call. You ask ChatGPT to "handle this ticket end-to-end" and trust whatever comes back.
  • A chain of LLM calls in a fixed sequence with no real branching or tool use. (This is sometimes called a "pipeline" or "chain" — it's a degenerate case of an agentic system.)

Our system is agentic because:

  1. It has 8 separate agents, each with a focused job
  2. The agents fan out and join (F1+F7 parallel; F3+F4 parallel)
  3. There's conditional routing (F6 only fires for on-site categories)
  4. Agents use tools (F2's 4 lookups, F8's notify_oncall, etc.)
  5. There's a human-in-the-loop gate (F5 never auto-sends)

How an agent actually runs (the inside view)

Here's what happens inside our F1 Triage agent when a ticket arrives:

agents/triage.py — simplified# 1. Safety check at the boundary assert_safe_input(state.description) # raises if jailbreak detected # 2. Build the prompt messages = [Message(role="user", content=state.description)] system_prompt = "You are a triage agent for a Belgian telecom..." # 3. Try the LLM path try: result = provider.complete( messages, schema=TriageOutput, tier=Tier.SMALL, system=system_prompt, ) except LLMProviderError: # 4. Fallback: keyword-based classifier, confidence capped at 0.5 result = _triage_fallback(state.description) # 5. Return a typed dict for LangGraph to merge into state return {"triage": result.model_dump()}

Notice what's NOT in here:

  • No tool calls (F1 is pure LLM, no external lookups)
  • No retries (the LLM provider's tool-use either succeeds or fails)
  • No "agent loop" — we don't let the LLM call itself in a circle

That last point is important. Some agentic frameworks let an agent decide for itself when to stop. We don't — our graph decides when each agent runs and when it stops. That's the "deterministic orchestration" commitment from concept 03.

Misconception to avoid

"An agent is autonomous — it just goes off and does things." That's one definition (think: an LLM-powered web browser that clicks around). In our system agents are much more constrained: each runs once per ticket, in a defined position in the graph, with a fixed output schema and a fixed tool allow-list. Bounded autonomy. That's the safer pattern for production.

Why 8 agents and not one big one?

You could imagine asking a single huge LLM call to do everything — classify the ticket, look up the partner, route it, draft a reply, all in one go. Why don't we?

Three reasons:

  1. Cost. A small model can classify a ticket accurately for $0.0001. Asking a frontier model to do everything end-to-end might cost $0.05 per ticket. That's 500× more expensive for less reliable results.
  2. Reliability. Each agent has one job, one schema, one fallback. If F4 Knowledge fails, the rest of the pipeline still runs and the engineer sees what wasn't fetched. A single end-to-end LLM call fails as one big blob — no partial recovery.
  3. Auditability. Each agent emits a log line with trace_id, latency, tokens. You can replay exactly what each one did, and the eval suite can score each in isolation. With one big call, you can't tell which step went wrong.

Quick check

A reviewer asks: "Couldn't you just call Claude Opus once with the whole ticket and let it figure it out?" — what's your answer?

You could, but you'd trade three things. Cost: small models classify accurately for <$0.001; Opus end-to-end costs ~500× more. Reliability: partial failures (one tool timing out) don't cleanly recover from a monolithic call. Auditability: the eval suite scores each agent independently — F1 100%, F3 100%, F4 93%; with one big call you can't tell which step went wrong. Specialist agents with structured outputs are the production-shaped answer; one big call is the prototype shape.
