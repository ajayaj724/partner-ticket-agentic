Multi-agent orchestration

Eight agents are nice, but who decides which one runs when? That's orchestration. We use LangGraph — a state machine designed for LLM workflows. Here's what it does and why it beats the alternatives.

The problem orchestration solves

You have 8 agents. A new ticket arrives. Questions to answer:

  • Which agent runs first?
  • Can two agents run in parallel?
  • What state does each agent see?
  • What happens if one fails — does the pipeline stop?
  • How do we trace a single ticket from start to finish?

You could answer all of this with a giant main.py full of if/else and threading code. Many teams do. It's called "vibe orchestration" and it's a nightmare to debug.

An orchestration framework lets you describe the topology declaratively, then it handles the threading, checkpointing, and tracing for you.

If you know AWS Step Functions

LangGraph is to Python what Step Functions is to AWS Lambdas. You describe states (= agents in our case) and edges (= transitions) in a graph, and the runtime executes them in order, in parallel where you say so, with built-in retry and observability. The domain is different (in-process Python vs. cloud-orchestrated Lambdas), but the architectural shape is the same — a declarative DAG you can read and reason about.

What LangGraph actually is

LangGraph is a Python library for building stateful, multi-step computational graphs. Think of it as networkx + threading + JSON-Schema validation, designed specifically for LLM-powered workflows.

You build a StateGraph:

Simplified version of our graph.pygraph = StateGraph(TicketState) # Register each agent as a node graph.add_node("triage", triage_node) graph.add_node("linker", linker_node) graph.add_node("enricher", enricher_node) graph.add_node("router", router_node) graph.add_node("knowledge", knowledge_node) graph.add_node("scheduler", scheduler_node) graph.add_node("drafter", drafter_node) # Wire the topology — parallel fan-out from START graph.add_edge(START, "triage") graph.add_edge(START, "linker") # Both join at the Enricher (LangGraph waits for both) graph.add_edge("triage", "enricher") graph.add_edge("linker", "enricher") # Enricher fans out to Router and Knowledge graph.add_edge("enricher", "router") graph.add_edge("enricher", "knowledge") # Conditional: scheduler only fires for on-site categories graph.add_conditional_edges( "route_decision", _route_decision, {"scheduler": "scheduler", "drafter": "drafter"}, ) graph.add_edge("drafter", END) runnable = graph.compile() # Run it result = runnable.invoke(initial_state)

The graph is declarative. Look at the edges, and you can read the topology without running anything. Look at the conditional, and you know exactly what decides between scheduler and drafter.

The three things LangGraph gives you

1. Parallel fan-out and join, free

When two edges leave the same node and rejoin somewhere downstream, LangGraph runs them in parallel and waits for both before continuing.

In our project, F1 Triage and F7 Linker fan out from START, both join at F2 Enricher. F2 fans out to F3 Router and F4 Knowledge, both join at the route-decision node. We didn't write a single Thread() or asyncio.gather(). LangGraph handles it.

2. Conditional routing

add_conditional_edges takes a function that returns a string, and a mapping of returned strings to next nodes. Our _route_decision looks at the triage category and returns either "scheduler" or "drafter".

That's how F6 Scheduler runs only for on-site-eligible tickets (appointment_request, provisioning, on-site circuit_down) and gets skipped otherwise.

3. A typed state object that's passed between nodes

Each agent's output gets merged into a single TicketState object. The next agent reads whatever it needs from there. No globals, no side-channel data, no "wait, did this field get set?".

The two big architectural commitments

Working with LangGraph forces you to commit to two principles that turn out to be the right ones anyway:

The graph decides routing, the LLM decides content

This is the single most important sentence in the whole project. Memorise it.

Why? Because LLMs are good at generating content for a specific structured task, but bad at deciding what to do next. If you let an LLM decide whether to call F6 Scheduler or skip it, the LLM might forget halfway through, or invent a step that doesn't exist, or call F6 twice. The graph doesn't forget anything. The graph is just code.

So the graph (deterministic Python) decides which agent runs when. The LLM (inside each agent) decides what to write. Clean split of responsibility.

State is explicit, not ambient

Every node reads from and writes to the TicketState object. There's no "what's the current ticket?" implicit context stored in a global. The state is the truth.

This means you can serialise the state, store it, replay it, diff it. That's the foundation of the audit trail and the "replay-from-trace" capability the design doc names.

Why LangGraph over the alternatives

Graph-based orchestration with deterministic tool-call control — the application decides routing, not the LLM. Built-in state checkpointing for crash recovery. Mature observability hooks. The Python ecosystem is strongest for retrieval and eval. Semantic Kernel is native to the Microsoft estate but less idiomatic outside .NET. OpenAI Assistants / Bedrock Agents are managed runtimes — tighter vendor coupling, weaker control over tool-call policy.

What about the alternatives?

Two questions the reviewer might ask:

"Why not write the orchestration yourself in plain Python?"

You could. ThreadPoolExecutor + a dict + some if/else. You'd reinvent: parallel join semantics, conditional routing, state merging, error propagation, checkpointing, retries, span tracing. And every team would reinvent them slightly differently. A library means the team's mental model maps to library terminology, which means faster onboarding and fewer surprises.

"Why not use Semantic Kernel or LlamaIndex Workflows?"

Both are reasonable. We picked LangGraph because:

  • Its graph model is explicit (you write edges; you don't infer them from decorators or class hierarchies)
  • It has checkpointing built in — pause and resume a graph mid-execution after a crash
  • The Python ecosystem around it (LangSmith for evals, LangChain for retrieval primitives) is the broadest
  • The 1.x stable release happened in 2025 — mature enough to defend in production

Misconception to avoid

"LangChain and LangGraph are the same thing." They're related but distinct. LangChain is a toolbox of building blocks (LLM wrappers, retrievers, parsers). LangGraph is a state-machine engine for composing those blocks into workflows. We use a tiny slice of LangChain (just BaseModel via Pydantic underneath) and most of LangGraph.

Quick check

Look at our graph topology. Why does F1 Triage run in parallel with F7 Linker, but F3 Router and F4 Knowledge run AFTER F2 Enricher?

Because of data dependency. F1 Triage classifies the ticket; F7 Linker searches for duplicate tickets. Neither depends on the other's output — they only need the raw ticket. So they run in parallel. F3 Router and F4 Knowledge both need the enriched context (partner profile, recent tickets, runbooks) that F2 produces — so they wait for F2. Parallel where independent, sequential where dependent. The topology is a literal map of the data-dependency graph.
