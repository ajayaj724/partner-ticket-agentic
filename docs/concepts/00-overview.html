What we built

Before the vocabulary, the application itself: what it does, what problem it solves, the 8 features, the demo workflow. Read this page and the rest of the series has context to hang on.

The shortest possible version

The application is a partner-ticketing platform for a Belgian telecom and its upstream fiber-installation partners, redesigned so that AI agents do the boring judgement work that humans used to do — and a human still approves anything that gets sent to the partner.

It's an open-source MIT-licensed reference implementation. It runs offline on a deterministic mock LLM (no API key needed). Anthropic and Ollama providers are opt-in. The whole demo fits in three commands.

The before-and-after

Before (the shipped project)

A web app + APIs for the telecom's operations team to file tickets with its upstream fiber-installation partners — organisations like BrusselsNet, Antwerp Cable, Liège Fibre — when monitoring or customer reports surfaced a fiber-line issue, and to coordinate the partner-side field-technician appointments that resolve it. CRUD on tickets + appointment management.

The flow: issue detected → ops files a ticket with the partner → a human on our side triages it, looks up context, picks the right partner-facing owner, finds the runbook, drafts the outbound message, sends it → partner schedules a technician → technician fixes the issue on-site → ticket closes. The shipped project did the data plumbing for all of that. AI was on the roadmap as a "next phase" but never got built.

After (this redesign)

Same domain shape from the outside — operations files a ticket, a partner technician eventually shows up and fixes it. Inside, the boring judgement work (triage, routing, runbook lookup, drafting) is now done by 8 specialist AI agents. The human only steps in at the end — to approve, edit, or reject the outgoing message to the partner.

If you know a hospital emergency room

Same shape: triage nurse sorts patients by urgency, receptionist pulls the chart, doctor is assigned, doctor consults medical guidelines, procedure scheduled if needed, diagnosis note drafted, senior doctor signs off before anything goes on the record, nurses periodically check admitted patients. Each step is a specialist; the senior is the gate. That's our system applied to telecom tickets instead of patients.

The 8 features (F1 - F8)

Each agent has one job. They fire in this rough order when a new ticket arrives:

The assembly line1. F1 Triage "circuit_down, critical urgency, CIRC-44781" 2. F7 Linker "this is a re-report of T-9001 (same partner)" 3. F2 Enricher partner profile + circuit state + 5 recent tickets + 3 runbooks 4. F3 Router queue=NOC-L2, assignee=u-noc-001, SLA=30 min 5. F4 Knowledge "runbook RB-001 is most relevant — here are 3 steps" 6. F6 Scheduler 3 proposed appointment slots, ranked by score (if on-site) 7. F5 Drafter drafts the message to the partner, requires_approval=true 8. HITL human clicks Approve / Edit / Reject (separate) F8 Watchdog every 5 min, scans all open tickets, pings on-call before any of them breach SLA

F1 and F7 run in parallel from the start. F3 and F4 run in parallel after F2. F6 is conditional — only fires for on-site categories (appointments, provisioning, on-site outages). F5 always runs last, with the HITL gate. F8 is a separate scheduled process, not part of the request/response chain.

Each command finishes in well under a second. The whole demo — including the web UI — fits inside a 3-minute walkthrough.

What the user sees in the web UI

The web UI is the most concrete way to grasp the system. Open http://127.0.0.1:8000 and you see:

  • Left pane: the LangGraph topology as an SVG diagram. As a ticket runs, the nodes light up in sequence: F1 + F7 in parallel, then F2, then F3 + F4 in parallel, then F6 (if applicable), then F5.
  • Right pane (Pipeline tab): per-agent cards appearing as each agent fires. Each card shows the Pydantic-validated output, latency, confidence, citations.
  • F5 card has the HITL gate: Approve / Edit / Reject buttons. The Edit button makes the draft body contentEditable; Approve flips a green badge.
  • Watchdog tab: one-click F8 scan; risk-band- coloured bars per at-risk ticket.
  • Safety tab: paste any text (or click "Fill jailbreak example"); the prompt-injection filter shows which regex pattern tripped, exact match highlighted.
  • Trace tab: the JSON-line audit log of the last run.

Screenshots of all five surfaces are committed in docs/screenshots/ in case the reader can't run anything live.

What's real in the demo and what's mocked

Honest scoping matters — and being explicit about it is the right shape for any technical review.

Real (production-shaped)

  • LangGraph orchestration — real StateGraph, real parallel fan-out, real conditional edges
  • Pydantic schemas on every agent output — free-text outputs are rejected
  • Tool allow-listing — typed ToolNotAllowedError on any out-of-list call
  • Prompt-injection filter at ingest — real regex, real rejection (exit code 4)
  • PII detection at ingest — email / phone / IBAN / IPv4
  • BM25 + FAISS hybrid retrieval over the runbook corpus
  • Idempotency keys on side-effecting tools (notify_oncall, escalate_to_manager)
  • Retry + circuit breaker on every tool call
  • Cost telemetry — pricing table against published Anthropic Jan 2026 rates; real numbers on --llm-provider anthropic
  • Per-tenant budget enforcement with 70/90/100% alerts
  • Anthropic prompt caching via cache_control: ephemeral
  • MCP server — any MCP client can connect
  • EU AI Act assessment document
  • Eval suite — precision/recall in CI

Mock / scoped for the demo

  • The LLM — the mock is deterministic if/elif rules. Real LLM via --llm-provider anthropic.
  • The downstream systems — CRM, inventory, calendar are in-memory stubs over the seeded JSON files. Production wires the real platforms.
  • The embedder — deterministic FNV-1a feature hashing. Production swaps one function for a real model (nomic-embed-text, Voyage, etc.).
  • The episodic store — SQLite at ~/.ptag/episodic.db. Production would be Postgres or DynamoDB.

Why this scoping is good engineering

The boundary between "real" and "mock" is the boundary at which production would swap one thing. Every mock is a single function or class with a clear interface; the rest of the system doesn't care which side of the boundary it's on. That's what "production-shaped reference implementation" means — it's not toy code, it's the real shape with the costly externals stubbed.

What this is NOT

To avoid setting wrong expectations:

  • It's not a SaaS. It's a single Python package that runs on your laptop. Production deployment is a next-phase exercise.
  • It's not training a model. The LLM is an external dependency (Anthropic API, Ollama server) or a mock. Nothing in this codebase trains anything.
  • It's not domain-bound to telecom. The 8 agents would work just as well for IT help-desk, insurance claims triage, or any B2B field-service ticketing flow. Telecom is the worked example.
  • It's not the production-ready version. It's the architectural skeleton with honest scoping. Things like real OIDC auth, full multi-tenancy, Postgres-backed episodic memory, OTLP-exporter OpenTelemetry pointed at your collector — all out of scope.

How it connects to the original project

The brief from the recruiter said: "If you did not [build agent workflows], provide a redesign of your project as if it used AI agents."

This codebase is exactly that redesign. The shipped project's domain (Belgian telecom partner-ticketing) and core entities (partners, circuits, runbooks, appointments) are preserved. What changes is the processing layer — instead of humans doing sort-route-lookup-draft, agents do it with a human gate at the outbound.

Slides 1-8 of the deck cover the shipped project. Slides 9-20 cover this redesign. Reviewers can dig into both.

Misconception to avoid

"This is a demo, so the architecture is also demo-grade." Not really. The orchestration, schemas, tool allow-listing, prompt-injection defence, PII detection, idempotency, retry, cost telemetry, budgets, MCP, audit trail, AI Act assessment are all production-shaped. The infrastructure cost of running this against real LLMs is a config change — flip --llm-provider, point it at a Postgres episodic store, point OTel at your collector. The mocks are at the edges; the core is real.

Quick check

Without looking back: what's the ONE thing the system does not let an agent do without human approval, and why?

Sending a message to a partner. Enforced by F5 always setting requires_approval=True on its output; the pipeline ends there until a human clicks Approve / Edit / Reject. Why: outbound communication is the highest-blast-radius surface — auto-sent messages could make SLA commitments the company can't keep, leak data, or contain hallucinated facts that erode partner trust. Auto-send is a non-starter for v1 in a regulated telecom context. This is the "human-in-the-loop" (HITL) commitment from slide 18 of the deck and concept 07 of this series.
