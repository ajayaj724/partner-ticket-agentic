RAG & retrieval

An LLM only knows what it was trained on. Retrieval-augmented generation is how we give it access to your private data — runbooks, ticket history, internal docs — at query time. The family of techniques behind it.

The problem RAG solves

You ask Claude: "Which runbook should I follow for a CIRC-44781 outage on a gold-tier partner?" Claude has never seen your runbooks. Its training data ended in early 2026, and it doesn't know about your internal telecom procedures.

Two bad options:

  1. Fine-tune the model on your runbooks. Expensive ($10k+), slow (days), and you have to redo it every time a runbook changes.
  2. Stuff every runbook into every prompt. Even if Claude's context window is 1M tokens, you'd burn $0.50 per ticket on input tokens alone, and the relevant content gets diluted.

The good option is retrieval-augmented generation (RAG): at query time, fetch only the relevant runbooks, paste them into the prompt, and let the LLM reason over them. Cheap, fast, and the runbooks can change without retraining anything.

If you know full-text search

RAG is "search-then-prompt". Imagine if every time a customer opened a support chat, you ran a search over your wiki, took the top 3 hits, and pasted them into the bot's prompt as context. That's RAG. The wrinkle is how you do the search — keyword? Semantic similarity? Hybrid? — and that's where all the technique lives.

The two retrieval techniques (and why we use both)

Technique 1 — BM25 (keyword search)

BM25 is a classic lexical search algorithm — the one ElasticSearch and OpenSearch use by default. It scores a document by how often the exact query terms appear in it, weighted by how rare those terms are across the whole corpus.

Query: "appointment reschedule request"

BM25 finds the runbook whose title is literally "Appointment reschedule request" because all three query words appear in the runbook text. Lexical match. Fast. Predictable. Works great when the user uses the same words as the docs.

Technique 2 — Dense (semantic) search via embeddings

An embedding is a fixed-length vector of numbers that represents the meaning of a piece of text. Two pieces of text that mean similar things end up with similar vectors, even if they don't share any words.

Conceptually"my circuit is dead" → [0.21, -0.14, 0.88, ..., 0.31] "the link is unreachable" → [0.19, -0.12, 0.85, ..., 0.33] "order pizza" → [-0.41, 0.62, -0.18, ..., 0.04] The first two vectors are CLOSE (cosine similarity ~0.97). The third is FAR (cosine similarity ~0.05).

Dense search works by computing the embedding of your query and finding the documents whose embeddings are closest to it. Works great when the user phrases things differently from the docs — "my circuit is dead" still matches a runbook titled "Partner-reported circuit outage" because both vectors point in similar directions in meaning-space.

Where do embeddings come from? An embedding model. Real production deployments use nomic-embed-text, voyage-3, or text-embedding-3-small. Our demo uses a deterministic hashing embedder (FNV-1a feature hashing) so the demo runs offline and the eval suite is reproducible — production swaps one function and the rest of the plumbing stays identical.

If you know OpenSearch

OpenSearch supports both — its classic match query is BM25, and its knn query is dense vector search. Modern stacks blend them ("hybrid search") because each catches what the other misses. We do the same thing in Python with FAISS for the dense index and a hand-rolled BM25 — but the architectural pattern is identical to what you'd deploy on OpenSearch in production.

Hybrid retrieval (what we actually use)

Why pick one when you can blend them?

  1. Run both BM25 and dense over the corpus.
  2. Normalise each set of scores to [0, 1] (min-max).
  3. Blend with a weight: alpha · dense + (1 - alpha) · bm25.
  4. Sort by the blended score. Take top-k.

With alpha = 0.5, both signals weigh equally. With alpha = 1.0, it's dense-only. With alpha = 0.0, it's BM25-only. You can tune per use case.

Our F4 Knowledge agent uses hybrid retrieval with alpha = 0.5 over the runbook corpus. It returns:

F4 output{ "top_runbook": { "runbook_id": "RB-001", "title": "Partner-reported circuit outage" }, "citation": "RB-001", # auditable — every suggestion cited "confidence": 0.494, "suggested_steps": [ "Confirm circuit status in inventory", "Run a latency probe; escalate to NOC if probe fails > 5 minutes", "Notify partner with status update once probe result is known" ] }

The "rerank" step (and why it matters)

Top-k retrieval gives you the rough candidates. But the ordering within those candidates may not be perfect. A cross-encoder reranker takes the top-k and re-scores each pair (query, candidate) with a more expensive model that looks at them together.

Cheap retrieval (BM25 + dense): top 20 candidates. Expensive rerank: top 3 to actually use.

Same idea as Google: cheap broad-index lookup gets ~thousands of pages, an expensive ranking model picks the 10 you see.

The two big design choices in RAG

1. Always cite

Every F4 output carries a citation field with the runbook ID. An engineer reading the suggested steps can click through to the source. No citation, no claim.

This is the antidote to LLM hallucination: if the model summarises a runbook incorrectly, the human can verify against the cited source. Without citations, RAG output is indistinguishable from hallucination.

2. Fall back below threshold

If the top hit's score is below 0.20, F4 returns fallback_reason="no high-confidence match" rather than handing the engineer a bad suggestion. This matches the design doc's safety contract: "return 'no high-confidence match' rather than guess".

Misconception to avoid

"RAG means feeding documents to ChatGPT." That's the surface. The work is in retrieval quality — how you chunk documents, what you embed, which similarity metric, hybrid vs. dense-only, rerank or not, confidence threshold. Bad retrieval = LLM gets irrelevant context = output looks correct but isn't. "Garbage in, fluent garbage out."

Quick check

A reviewer asks: "Why hybrid retrieval instead of just dense embeddings?"

Because dense and BM25 catch different things. Dense excels when the query and the doc use different words for the same idea ("my circuit is dead" vs. "partner-reported outage"). BM25 excels when the query uses exact terms from the doc (a circuit ID, a queue name, a product code). Either alone has blind spots. Blending min-max-normalised scores with a tunable alpha gives you both signals; for our runbook corpus we use alpha=0.5. The eval suite scores F4 at 0.933 top-1 — a real embedder in production (Voyage, nomic-embed-text) would push that toward 0.95+.
