Now that you know the concepts and what the application does, here's a file-by-file tour. Designed so you can answer "where would I find X in the code?" for anything the panel asks about.

Repository rootpartner-ticket-agentic/ ├── src/partner_ticket_agentic/ ← all the application code ├── tests/ ← 193 pytest tests ├── evals/ ← 5 JSONL golden sets ├── docs/ ← design spec + guides + this concept series ├── data/ ← seed JSON (partners, runbooks, sample tickets) ├── config/ ← approved_models.yaml, budgets.yaml ├── Presentation/ ← the PPT for the panel ├── pyproject.toml ← deps, build, lint, test config ├── uv.lock ← reproducible dep versions ├── .github/workflows/ci.yml ← lint + test + eval on every push ├── README.md ← architect-grade entry-point └── LICENSE ← MIT (Ajay Antony, 2026)

Four things to notice:

  • src/ layout — the package is under src/ rather than at the root. Standard Python convention for libraries that want to be installable cleanly.
  • tests/, evals/, docs/ as peers — same level as src. Says "tests and evals are first-class artefacts, not afterthoughts."
  • config/ for runtime knobs — separate from code. Approved models and per-tenant budgets are config, not hardcode.
  • data/ holds the demo's "world" — partners, runbooks, sample tickets. Production wires the real platforms; the demo uses these seeded JSON files.

How to read the codebase (in order)

Don't read every file. Read these five in this order — that's enough to defend any panel question:

1. graph.py (~130 lines)

The whole topology in code form. StateGraph(TicketState), add_node for each agent, add_edge for transitions, add_conditional_edges for F6's branch. The run_pipeline function ties trace IDs, log context, cost ledger, and budget state to one ticket flow.

If you read one file end-to-end, this is it. It IS the architecture.

2. agents/triage.py (~300 lines)

Smallest and cleanest agent. Read it to see the agent pattern in code: schema definition (TriageOutput), mock-LLM rule (_triage_rule), fallback (_triage_fallback), main entry (run_triage), LangGraph node wrapper (triage_node). Every other agent follows the same shape — read this once, you've read them all.

3. providers/mock.py (~150 lines)

The deterministic LLM stand-in. Look at the rule registry (_rules), the dispatch logic (complete), and the cost/budget hooks. CLAUDE.md is explicit: "deterministic if/elif rules a reviewer can read and predict — no random, no hashing, no time-based branching." This file enforces that contract.

4. safety.py (~130 lines)

The two safety primitives in one file: ToolAllowList with the typed ToolNotAllowedError, then detect_prompt_injection with the regex patterns, then detect_pii + redact_pii_for_logging. Compact and auditable.

5. tools/policy.py (~150 lines)

Retry + circuit breaker. RetryPolicy with exponential backoff. CircuitBreaker as a three-state FSM (CLOSED / OPEN / HALF_OPEN). BreakerRegistry with a process-wide default. Read once and you understand the "fail closed, recover gracefully" posture.

If you only have 15 minutes

Read graph.py (the architecture in code) and agents/triage.py (one agent end-to-end). That's the 80/20. The other three above are nice-to-have for depth questions on safety, providers, or tools.

How to extend it

If a panellist asks "how would you add a ninth agent?" the answer is a concrete walkthrough:

  1. Create src/partner_ticket_agentic/agents/foo.py with:
    • A Pydantic FooOutput schema
    • A mock-LLM rule registered against the schema
    • A run_foo function
    • A foo_node LangGraph wrapper
  2. Add it to graph.py with the right edges (parallel? sequential? conditional?).
  3. If it needs tools, register them in tools/ and add to the agent's ToolAllowList.
  4. Add a JSONL eval set under evals/ and wire it into the runner.
  5. Write unit tests in tests/test_foo.py + integration tests in tests/test_graph.py.
  6. Update docs/DESIGN.md §3 with the new feature's spec.

Every existing agent follows this pattern. agents/triage.py is the template.

For the panel

If they ask "how would your team add a new feature without breaking the existing ones?", you can show them: the module-per-agent structure, the schema-first I/O contract, the per-agent allow-list, and the eval suite gates in CI mean a new agent is additive — it can't accidentally break F1-F8. That's what production- shape means at the team level, not just the code level.

The two artefacts inside the repo that aren't code

  1. docs/DESIGN.md — the authoritative spec. ~270 lines. Read first if you want the architectural reasoning rather than the implementation details. Section numbers in the spec are referenced throughout the code comments.
  2. docs/AI_ACT_ASSESSMENT.md — the limited-risk classification document. Concept 07 explains why this matters; this is the doc that proves we've thought about it.

Plus the doc set you're reading: docs/concepts/ (this series), docs/learn.html (study guide), and docs/demo-guide.html (talk-track).

Misconception to avoid

"The code is the documentation." For a 2-week panel project maybe; for anything bigger, the documents matter as much as the code. The design doc explains the why the code can't. The eval golden sets capture intent the tests can't. The AI Act assessment makes regulatory thinking visible to a reviewer. Code, docs, evals, and assessment all ship as one artefact.

Quick check

The panel asks: "Show me where in the code I'd find your prompt-injection defence." What do you point at, and what does it do?

src/partner_ticket_agentic/safety.py — specifically detect_prompt_injection(text) and assert_safe_input(text). The first function returns a list of InjectionFinding objects (each carrying the matched regex + the substring). The second wraps the first and raises SafetyError if any pattern matches — used at the CLI boundary in cli.py for the --inject flag, and at the ingest boundary in graph.py before the pipeline starts. Three layers of defence: this regex filter at ingest, the system-prompt anchoring inside each agent, and the per-agent tool allow-listing. Defence in depth.
