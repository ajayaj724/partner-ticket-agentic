Concept 06 of 08

Three memory tiers

Agents need state. But state spans different time-scales and different scopes. We keep three tiers — working, episodic, long-term — and we never conflate them. Here's why.

Why "memory" is a concept at all

A regular HTTP API is stateless — each request stands alone. An agent isn't. The agents in our pipeline have to share state across their run (F2's output feeds F3 and F4). The partner's history matters across tickets (this is the 12th circuit-down report this month). Organisational knowledge matters across all partners (the runbook for circuit outages hasn't changed in a year).

Three different time-scales. Three different scopes. Three different lifetimes. That's why we keep three separate stores.

The three tiers

Tier Scope Lifetime Store
Working One ticket flow Seconds (in-memory) LangGraph TicketState
Episodic Per partner Across runs, persisted SQLite at ~/.ptag/episodic.db
Long-term Org-wide Indefinite FAISS + SQLite

Working memory — the LangGraph state

Working memory is the in-memory state object passed between nodes in the graph. It lives for the duration of one ticket flow. F1's output lands in it. F2 reads from it and writes enrichment. F5 reads everything before drafting.

memory/working.pyclass TicketState(BaseModel): ticket_id: str partner_id: str subject: str description: str triage: dict | None = None enrichment: dict | None = None routing: dict | None = None knowledge: dict | None = None related: dict | None = None schedule: dict | None = None draft: dict | None = None trace_id: str cost: dict | None = None pii_findings: list[dict] = []

When the pipeline finishes, the state is discarded. The trace log captures what happened; the state itself doesn't survive.

Episodic memory — per-partner, persisted

"Episodic" comes from cognitive science — memories of specific events (the time you went to the concert; the time you had pasta in Rome). Our agents' episodic memory is per-partner: a compact summary of past ticket flows for partner P-1001.

F2 Enricher's ticket_history_recent tool reads this tier. The Watchdog (F8) can read it to spot partners who have had multiple incidents in a week. The Linker (F7) uses it (via similarity over recent tickets from the same partner) to detect duplicates.

memory/episodic.pystore = EpisodicStore("~/.ptag/episodic.db") # Write on ticket completion store.record( partner_id="P-1001", ticket_id="T-A", category="circuit_down", urgency="critical", summary="CIRC-44781 outage; resolved in 47m", ) # Read in F2 Enricher recent = store.recent("P-1001", limit=10)

SQLite as the demo backend. Production would be Postgres or DynamoDB depending on the platform's existing stack.

Long-term memory — org-wide knowledge

Long-term memory holds organisational knowledge that's shared across all partners. In our system that's the runbook corpus, indexed two ways (FAISS for dense, BM25 for lexical) plus structured facts (categories, queue mappings, SLA policies) in SQLite.

F4 Knowledge reads this tier (the runbook search; see concept 05). F3 Router reads the SLA policy table from here.

If you know your shipped project's data tier

Your shipped project split DynamoDB (single-key lookups) and OpenSearch (faceted search) because the access patterns were different. Same thinking here — the three memory tiers exist because the access pattern is different per tier: in-memory for in-flight state, indexed-by-partner for episodic, semantic + lexical for long-term.

The rule: never conflate the tiers

The design doc names this explicitly: "three memory tiers, never conflated". Why?

It's tempting to dump everything into one big "memory" store — a single vector DB with partner_id as metadata, working state mixed in, runbooks alongside ticket summaries. Some teams do this with Pinecone or Weaviate and call it "the agent's memory".

The problem: when you conflate tiers, you lose:

  • Access control. The Enricher should be able to read partner P-1001's episodic memory but not P-1002's. Tenant scoping is much easier when episodic is a separate table keyed by partner_id.
  • Lifecycle control. GDPR right-to-erasure means we have to purge per-partner data on offboarding. If partner data is mixed with org-wide knowledge, you have to diff every embedding. If episodic is a separate tier, you just DELETE FROM episodic WHERE partner_id = ?.
  • Cost control. Working memory is free (in-process). Episodic is cheap (a SQLite table). Long-term is the expensive one (FAISS index, embedding model calls). Mixing them means you pay long-term costs for working-memory operations.
  • Debuggability. When something goes wrong, knowing which tier the bad data came from is half the diagnosis. "The triage label is wrong" → working memory. "We forgot the partner's tier" → episodic. "We cited the wrong runbook" → long-term. Three tiers, three different root causes.

Misconception to avoid

"Agentic systems have a memory" — singular. They don't. Production-grade systems have at minimum the three tiers above; many have more (session memory, user-preference memory, tool-result cache). Conflating them is the architectural mistake most teams make in their first agentic deployment. The discipline of separation is what makes the system maintainable past month three.

GDPR right-to-erasure — a concrete example

A partner offboards. GDPR says we have to purge everything personal about them within 30 days. With three tiers, the flow is clean:

  1. Working memory: nothing to do — it never persisted across runs.
  2. Episodic memory: DELETE FROM episodic_entries WHERE partner_id = 'P-1001'. One SQL statement.
  3. Long-term memory: runbooks are org-wide, not partner-specific — nothing to purge. Structured facts: same.

If you'd dumped partner-specific data into a shared vector store, step 3 would be: re-embed every document, diff the index, remove vectors. Expensive and error-prone.

Quick check

A reviewer asks: "Where do you store the system's understanding of a specific partner's preferences and history — and why there?"

Episodic memory — SQLite, keyed by partner_id, persisted across runs. Per-partner because the scope is one tenant; we don't want F2 Enricher accidentally surfacing P-1002's preferences when handling P-1001's ticket. SQLite because the access pattern is "give me the last N rows for this partner" — that's an indexed scan, not a vector search. We keep it separate from working memory (in-process) and long-term (org-wide FAISS) because the three have different access patterns, different lifecycles, and different access-control requirements.
