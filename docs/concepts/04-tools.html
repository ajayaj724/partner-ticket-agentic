Tools & the security primitive

An agent's "tools" are the functions it can call to fetch data or take action in the outside world. Tools are also the security boundary of an agentic system. Get this one right and a lot of safety follows.

What "tool" means in our world

A tool is a regular Python function — typed inputs, typed outputs — that an agent can invoke during its run. Examples from our project:

  • crm_lookup_partner(partner_id) → PartnerProfile
  • inventory_lookup_circuit(circuit_id) → CircuitState
  • ticket_history_recent(partner_id, n=5) → list[Ticket]
  • runbook_search(query) → list[RunbookHit]
  • notify_oncall(ticket_id, queue, idempotency_key, message)

From the LLM's perspective, a tool is a structured action it can ask the runtime to perform on its behalf. The runtime (our Python code) actually executes the function and returns the result back to the LLM — or in our case, to the agent that wraps the LLM.

If you know REST APIs

Think of an agent as a service that calls downstream APIs. Each "tool" is a downstream endpoint. The difference is that an agent decides which endpoints to call based on its input — like a Quarkus Lambda that conditionally calls system A, B, or C depending on the request. Tools are the agent's downstream surface.

Why tools are the security primitive

Without tools, an LLM is sandboxed — it can only produce text. The worst it can do is say something wrong.

With tools, the LLM can reach into your systems. Read a partner's data. Send an email. Cancel an appointment. The instant you wire up tools, you've moved from "LLM as text generator" to "LLM as actor". Now the question becomes: what's it allowed to do?

Our answer: each agent has a per-agent allow-list of tools, enforced in code as a typed exception, not a convention.

What that looks like

F2 Enricher's allow-list# In agents/enricher.py ALLOW = ToolAllowList.of( "enricher", "crm_lookup_partner", "inventory_lookup_circuit", "ticket_history_recent", "runbook_search", ) dispatcher = ToolDispatcher(allow_list=ALLOW) # F2 calls only those four. Anything else raises: try: dispatcher.call("notify_oncall", ...) # F2 doesn't have this except ToolNotAllowedError: # Caught at the boundary. Audit log entry. Hard-fail. ...

If F2 ever tried to call notify_oncall (an F8 tool), ToolNotAllowedError is raised — a typed exception that bubbles up the stack with the agent name, the tool name, and the actual allowed list. The audit log captures it. The pipeline stops.

If you know AWS IAM

Per-agent tool allow-listing is the LLM equivalent of an IAM role policy. Each agent has an IAM role; each tool is an Action. An agent can only invoke actions inside its policy. ToolNotAllowedError is the equivalent of AccessDeniedException — and just like IAM, the safest default is deny by default, allow explicitly.

Two other tool-design choices that matter

1. Idempotency keys on side-effecting tools

Most tools just read — looking up a partner, fetching a runbook. Reading is naturally idempotent (you can do it twice without side effects).

Some tools do things — notify_oncall sends a Slack message; escalate_to_manager opens an incident. If F8 Watchdog scans every 5 minutes and a ticket is at risk for 30 minutes, we'd notify on-call 6 times — once per scan. That's spammy and irresponsible.

Fix: every side-effecting tool takes an idempotency key as a required argument. Re-calling with the same key returns deduplicated=True without firing the side effect:

Idempotency in tools/oncall.py@register_tool("notify_oncall", description="...") def notify_oncall(*, ticket_id, queue, idempotency_key, message): if idempotency_key in _Idempotency.seen: # We already sent this. Don't spam. return NotificationResult( ticket_id=ticket_id, channel=f"oncall:{queue}", delivered=False, deduplicated=True, ... ) _Idempotency.seen.add(idempotency_key) # ... actually fire the notification return NotificationResult(... delivered=True ...)

F8 uses a stable key like f"breach-watch:{ticket_id}" so all 6 scans share the same key — only the first one fires.

If you know Stripe

This is exactly Stripe's Idempotency-Key header. Same pattern, different domain. Stripe uses it so a payment retry doesn't charge the customer twice; we use it so a watchdog retry doesn't spam on-call twice.

2. Retry with backoff + circuit breaker

Tools call downstream systems. Downstream systems fail transiently. Our ToolDispatcher wraps every call with:

  • Retry with exponential backoff — up to max_retries attempts, with delays of 50ms, 100ms, 200ms.
  • Circuit breaker — if a tool fails N times in a row, the breaker opens; subsequent calls fast-fail without invoking the tool. After a cooldown, one probe attempt determines whether the breaker closes or stays open.

This is a classic infrastructure pattern (Hystrix from Netflix, Resilience4j on the JVM). The point is to fail fast when a dependency is down rather than hammering it with retries.

Surfacing tools via MCP

MCP is the Model Context Protocol — Anthropic's open standard for exposing tools to LLM clients. Released 2024, widely adopted by 2026.

Our entire tool registry can be served as an MCP server with one command:

Run it$ python -m partner_ticket_agentic --mcp Booting MCP server on stdio (Ctrl-C to stop)

Once running, any MCP-aware client — Claude Desktop, the mcp CLI, another agent in another project — can discover and invoke our tools over the protocol. This is the slide 13 commitment in the deck: "surfaced via MCP for reuse across agents".

Why panels care about MCP

MCP is the 2026 industry signal. If your agentic system can't speak MCP, it's a closed silo — only your own agents can use your tools. If it can speak MCP, it's interoperable with the broader ecosystem. Saying "we surface the tool registry via MCP" tells the panel you follow industry standards, not just bespoke conventions.

The four tool-design properties to remember

  1. Typed I/O. Inputs and outputs are Pydantic models. The dispatcher rejects misshapen calls.
  2. Per-agent allow-list. Each agent declares its tool surface explicitly. ToolNotAllowedError is a typed gate, not a convention.
  3. Idempotent or idempotency-keyed. Read-only tools are naturally idempotent; side-effecting tools take an idempotency key so retries are safe.
  4. Retry + circuit-breaker at the dispatcher level. Tools don't worry about retry logic; the runtime handles it.

Misconception to avoid

"The LLM decides which tools to call." In our system the LLM decides what to write; the agent (Python code) decides which tools to call. The allow-list isn't enforced on the LLM — it's enforced on the agent's dispatcher. The LLM can't reach a tool that isn't in the allow-list because it can't call any tool directly; only the agent's Python code can.

Quick check

The panel asks: "What happens if F2 Enricher's crm_lookup_partner tool calls a downstream that's timing out 50% of the time?"

Three things in sequence. (1) Retry with exponential backoff at the dispatcher — first attempt fails, second try after 50ms, third after 100ms. (2) If retries exhaust, the tool raises ToolError and that section of F2's enrichment lands in unavailable=["crm"]; the engineer sees what was missing rather than a partial picture. (3) After enough consecutive failures the circuit breaker opens — subsequent calls fast-fail without touching the downstream, giving it room to recover. Production-grade behaviour without the agent code knowing any of this — the dispatcher handles it.
